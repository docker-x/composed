# Quick Start

This guide walks through creating a stack with a Helm chart and a plain Docker
image, building the final compose file, and starting everything. By the end you
will have LiteLLM (an AI gateway) backed by Postgres, running on plain Docker.

## 1. Initialize a project

Create a new directory and initialize it:

```bash
mkdir my-stack && cd my-stack
composed init --project my-stack
```

This creates a `composed.yaml` file with the project name:

```yaml
name: my-stack

services: {}
```

## 2. Add a Helm chart

Add the LiteLLM Helm chart from its OCI registry, pinning the image tag:

```bash
composed add oci://docker.litellm.ai/berriai/litellm-helm --set image.tag=main-stable
```

`composed add` probes the OCI manifest, detects that the artifact is a Helm
chart (not a container image), and writes the service entry with an `x-helm`
block. The service name is derived automatically from the chart name.

## 3. Add a Docker image

Add a Postgres instance as a plain compose service:

```bash
composed add postgres:15-alpine --port 5432:5432 --env POSTGRES_PASSWORD=secret
```

Because the source has no OCI chart metadata and no `Chart.yaml`, it is treated
as a regular Docker image. The `--port` and `--env` flags map directly to
compose `ports:` and `environment:` fields.

## 4. Review the config file

After both `add` commands, `composed.yaml` looks like this:

```yaml
name: my-stack

services:
  litellm-helm:
    x-helm:
      chart: oci://docker.litellm.ai/berriai/litellm-helm
      values:
        image:
          tag: main-stable
    depends_on:
      - postgres

  postgres:
    image: postgres:15-alpine
    environment:
      POSTGRES_PASSWORD: secret
    ports:
      - "5432:5432"
```

A few things to note:

- This is a standard Docker Compose file. The only non-standard part is the
  `x-helm` block under `litellm-helm`.
- Plain services like `postgres` work with `docker compose up` right now --
  Docker Compose ignores `x-` fields it does not recognize.
- Services with `x-helm` need `composed build` to resolve them into real
  compose services.

You can edit this file by hand at any time. Add `depends_on`, volumes,
environment variables, or any other compose field.

## 5. Build the compose file

Run the build step:

```bash
composed build
```

This runs the full pipeline:

1. **Helm render** -- pulls the OCI chart, runs `helm template` with the
   inline values (`image.tag=main-stable`).
2. **K8s parse** -- splits the rendered YAML into individual Kubernetes
   objects (Deployments, Services, ConfigMaps, Secrets, PVCs).
3. **Translate** -- converts each K8s resource into its Compose equivalent.
   Deployments become services, PVCs become named volumes, ConfigMaps and
   Secrets become environment variables, probes become healthchecks.
4. **Merge** -- combines the translated chart services with the plain
   `postgres` service into one file.
5. **Emit** -- writes the final `docker-compose.yaml`.

The output looks something like this:

```
composed: rendering litellm-helm (oci://docker.litellm.ai/berriai/litellm-helm)
composed: translated 2 Deployments, 1 Service, 1 ConfigMap, 1 Secret
composed: skipped 2 ServiceAccount, 1 NetworkPolicy (no compose equivalent)
composed: merged 3 services into docker-compose.yaml
```

The generated `docker-compose.yaml` contains only standard compose syntax --
no `x-` extensions, no Helm references. You can inspect it:

```bash
cat docker-compose.yaml
```

```yaml
# Generated by composed from my-stack
# Source: composed.yaml

name: my-stack

services:
  litellm:
    image: ghcr.io/berriai/litellm:main-stable
    environment:
      PORT: "4000"
      LITELLM_MASTER_KEY: "sk-1234"
    ports:
      - "4000:4000"
    healthcheck:
      test: ["CMD", "wget", "-q", "--spider", "http://localhost:4000/health"]
      interval: 10s
      timeout: 5s
      retries: 3
      start_period: 15s
    depends_on:
      - postgres

  postgres:
    image: postgres:15-alpine
    environment:
      POSTGRES_PASSWORD: secret
    ports:
      - "5432:5432"
```

The exact output depends on what the Helm chart templates produce. Resource
kinds that have no compose equivalent (ServiceAccount, RBAC, NetworkPolicy)
are skipped with a warning.

## 6. Start the stack

Bring everything up:

```bash
composed up
```

This runs `composed build` (if needed) and then `docker compose up -d` against
the generated file. You will see the normal Docker Compose output:

```
[+] Running 3/3
 - Network my-stack_default   Created
 - Container my-stack-postgres-1   Started
 - Container my-stack-litellm-1    Started
```

Check that the containers are running:

```bash
docker ps --filter label=com.composed.project=my-stack
```

## 7. Stop the stack

When you are done:

```bash
composed down
```

This runs `docker compose down`, stopping and removing the containers.

## What to do next

- Edit `composed.yaml` to add more services, change values, or wire up
  `x-exports` for cross-service references.
- Run `composed init --helm-values` to scaffold the full Helm values file for
  each chart, so you can customize every option.
- Look at the [examples/](../../examples/) directory for more realistic stacks,
  including shared Postgres with schema isolation and multi-chart setups.
- Read the [Config File](../guide/config-file.md) guide for the full reference
  on `x-helm`, `x-compose-file`, and `x-exports`.
